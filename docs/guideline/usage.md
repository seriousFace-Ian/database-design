# DB Design 使用说明

> 适用版本：Phase 8（2026-06-05 交付，含表达式索引、IDENTITY 列、EXCLUDE 排斥约束）
> 受众：第一次打开本工具的数据库设计者 / 后端工程师
> 阅读顺序：自上而下；前 3 节是必读，后面按需查阅

本工具是一个**完全在本地运行**的 PostgreSQL 数据库设计器。可视化拼好表结构后，既能导出 `.sql` 文件、也能直接把 DDL 执行到本地 PG，还能反向把已有库导回成可编辑的项目。所有项目状态以 `.dbdesign.json` 保存到本地磁盘，或以一行 JSONB 嵌入目标库的 `__dbdesign` 表（两种通道可共存）。

---

## 1. 启动与首屏

```bash
npm install              # 仅首次
npm run dev              # 同时拉起前端 (5173) 和后端 (3001)
```

打开 `http://localhost:5173` 进入主界面，布局如下：

```
┌─────────────────────────────────────────────────────────────┐
│  顶部工具栏                                                 │ ← 新建/打开/保存/视图切换/连接/导入/执行
├──────────┬──────────────────────────────────────────────────┤
│  侧边栏   │                                                  │
│  - 表列表 │            主工作区                              │
│  - ENUM   │   设计器 / 关系图 二选一                         │
└──────────┴──────────────────────────────────────────────────┘
```

**主要状态指示**：
- 工具栏右侧出现橙色 `未保存` Tag → 有改动未落盘。
- 连接图标：灰=未连接、绿=已连接、红=连接失败。
- 右下角 Toast 显示操作结果（保存成功、SQL 执行条数等）。

后端没有任何持久化，仅作为数据库代理；前端把连接信息（含密码）放在 `sessionStorage`，关闭标签即清空。

## 2. 项目管理

项目对应一个 `ProjectFile`（JSON），含 `name / description / enums / tables / diagramLayout`。

| 操作 | 入口 | 落地位置 |
|------|------|----------|
| 新建项目 | 工具栏「新建」 | 内存（提醒及时保存） |
| 打开本地 JSON | 工具栏「打开」 | 选 `.dbdesign.json` 文件 |
| 保存到本地 | 工具栏「保存」 / `Ctrl+S` | 覆盖原文件；新项目走「另存为」 |
| 存库 | 工具栏「存库」（需连接） | 目标库的 `__dbdesign` 表（一行 JSONB） |
| 读库 | 工具栏「读库」（需连接） | 把 `__dbdesign` 拉回前端，覆盖当前项目（带确认） |

**两种持久化通道并存**：
- **本地 JSON**：离线友好，可放进 git，团队评审简单。
- **DB-embedded**：项目跟着数据库走，做迁移 / 拷库时无需另外维护 `.dbdesign.json`。

> 推荐组合：日常用本地 JSON 配合 git；上线前用「存库」把"权威设计"写进目标库，运维人员后续 `读库 → 改 → 执行 → 存库` 形成闭环。

打开旧 `.dbdesign.json`（Phase 7 之前的索引格式）会被自动迁移为新结构（`fieldIds → columns`），文件结构在下次保存时升级。

## 3. 数据库连接

`工具栏 → 连接` 弹出连接面板：

| 字段 | 说明 |
|------|------|
| Host / Port | 默认 `localhost:5432` |
| Database | 目标库名（区分大小写）|
| Username / Password | PG 角色及密码 |
| SSL | 勾选后用 `ssl: true` 建池 |

**测试连接** → 后端建一次性 `pg.Pool`，跑 `SELECT version()`，返回成功后保留连接信息。**断开** 会清空 `sessionStorage` 并刷新页面。

> 服务端**不会**持久化连接信息。每个 `/api/*` 请求都带上凭据，调用结束 `pool.end()` 立即释放。

## 4. 设计器视图

### 4.1 表与字段

- **左侧表列表**：`+ 新建表` 创建空表；单击切换当前表；右键 / 删除图标删表（自动清理其它表中指向它的外键）。
- **中央 Tabs**：`字段`、`约束`、`索引`、`表信息`。
- **「添加字段」按钮** / 表脚的 `+ 添加字段`：追加一行；可拖动行手柄 (`⋮⋮`) 调整顺序。
- **「审计字段」按钮**：一键插入 `created_at` / `updated_at` / `deleted_at` / `created_by` 四列（同名字段会跳过）。

字段表列说明：

| 列 | 编辑方式 |
|----|----------|
| 字段名 | 行内输入 |
| 类型 | 分组下拉（整数 / 文本 / 数值 / 日期时间 / 其他 / ENUM）；VARCHAR/CHAR 出现长度框；NUMERIC 出现精度/标度框 |
| 可空 / PK / 唯一 | 复选框；勾 PK 会自动取消「可空」 |
| 默认值 | 输入框 + 右侧 ⚡ 预设下拉（详见 §4.3）|
| 注释 | 行内输入 |
| 操作 | `CHECK / IDENTITY 配置`、`外键`、`删除` 三个图标 |

### 4.2 IDENTITY（PG10+ 自增列）

**两个前置条件**，缺一不可：

1. 字段类型必须是 `SMALLINT` / `INTEGER` / `BIGINT` —— 其他类型下 IDENTITY 单选**不会显示**（这是 PG 自身的约束，不是 UI 限制）。
2. 入口是字段行**末尾**的 ✔ 图标（与 CHECK 复用同一个 Modal，tooltip 显示「列约束（CHECK / IDENTITY）」）。

打开 Modal 后底部出现 `IDENTITY（自增列）` 单选：

- `无（普通列）`
- `GENERATED BY DEFAULT AS IDENTITY`（推荐，可手动覆盖）
- `GENERATED ALWAYS AS IDENTITY`（严格，禁止手动赋值）

选 IDENTITY 后系统会自动清空该字段的「默认值」（PG 不允许两者共存）；默认值输入框也会被禁用。

与 `SERIAL / BIGSERIAL` 的关系：`SERIAL` 仍在类型下拉中作为"旧式快捷"。新项目推荐 `BIGINT + IDENTITY`。

> 如果你给一个 `BIGINT` 字段开了 IDENTITY、又把类型改回 `VARCHAR`，UI 不会自动清掉 `field.identity`；下次再点开 Modal 时也看不到单选（因为类型已不合规）。SQL 生成阶段会照样输出 `GENERATED ... AS IDENTITY` 然后被 PG 报错。这是已知边界，需在改类型时手动先把 IDENTITY 关掉，或在导入/编辑阶段保持一致。

SQL 输出示例：

```sql
CREATE TABLE upload_tasks (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  ...
);
```

### 4.3 默认值预设下拉

默认值输入框右侧的 ⚡ 按钮按字段类型智能排序：

| 类型敏感匹配 | 预设组排在最前 |
|--------------|----------------|
| TIMESTAMP / TIMESTAMPTZ / DATE / TIME | 时间戳：`now()` / `CURRENT_TIMESTAMP` |
| UUID | UUID：`gen_random_uuid()` / `uuid_generate_v4()` |
| BOOLEAN | 布尔：`true` / `false` |
| 数值类 | 数值：`0` / `1` |
| VARCHAR / TEXT / CHAR | 字符串：`''` |
| JSON / JSONB | JSON：`'{}'` / `'[]'` |
| 全部 | 序列：`nextval('seq_name')`（占位，需用户改名） |

预设是"填入后可手改"，不强制锁死。

### 4.4 外键

点击字段行的 🔗 图标 → 外键 Modal：

- 选「引用表」→「引用字段」（仅显示对方表的主键和唯一字段）。
- `ON DELETE` / `ON UPDATE`：`NO ACTION / RESTRICT / CASCADE / SET NULL / SET DEFAULT`。
- 约束名留空时按 `fk_<table>_<col>` 自动生成；自定义后下次同步会按名匹配。
- 已配置外键的字段在表节点和字段表上以蓝色 🔗 标记。

> SQL 生成阶段 FK 总以独立 `ALTER TABLE ... ADD CONSTRAINT` 语句发出，且**在所有 `CREATE TABLE` 之后**，因此循环外键也能落库。

## 5. ENUM 类型

侧边栏「ENUM」区域：

- `+ 新建 ENUM` → 弹窗输入 `schema.name` 和值列表（每行一个值）。
- 添加完后，字段「类型」下拉「自定义枚举」组里就能选到 `schema.name`。
- 删除 ENUM：所有引用它的字段自动回落为 `TEXT`。

> SQL 顺序：`CREATE TYPE` 永远先于 `CREATE TABLE`，避免列引用未声明类型。

## 6. 表级约束（UNIQUE / CHECK / EXCLUDE）

中央 Tabs 切换到「约束」面板。顶部三个按钮：

```
[+ 添加 UNIQUE]   [+ 添加 CHECK]   [+ 添加 EXCLUDE]
```

每条约束以独立卡片显示，左侧 Tag 颜色：

| Tag | 含义 |
|-----|------|
| 🟢 UNIQUE | 组合唯一约束 |
| 🟣 CHECK | 跨列 CHECK 表达式 |
| 🟠 EXCLUDE | 排斥约束（Phase 8） |

### 6.1 UNIQUE

- **至少 2 个字段**：单列唯一请在字段表勾「唯一」。
- 约束名留空 → `uq_<table>_<col1>_<col2>`。

### 6.2 CHECK（跨列）

- 输入裸表达式即可，无需写 `CHECK` 与外层括号。例如：`start_date < end_date`、`total = price * quantity`。
- 可选"关联字段"仅作 UI 元信息，与 SQL 无关。
- 约束名留空 → `chk_<table>_<短哈希>`。

> 单列 CHECK 请用字段行末尾的 ✔ 图标在 IDENTITY 配置 Modal 里填。

### 6.3 EXCLUDE 排斥约束（Phase 8）

**典型用例**：会议室预订防重叠、价格档位防交叉、IP 段防覆盖等。

打开 Modal，参数：

| 字段 | 说明 |
|------|------|
| 约束名 | 留空 → `ex_<table>_<col1>_<col2>` |
| 索引方法 | `GIST` 默认；可选 `SPGIST / BTREE / HASH` |
| 元素 | 至少 1 行；每行：`字段/表达式 + WITH 操作符` |
| WHERE 谓词 | 可选，写裸表达式（不带 WHERE 关键字） |
| 高级 → DEFERRABLE / INITIALLY DEFERRED | 复选框，可选 |

元素操作符常用预设：`=`、`<>`、`&&`、`@>`、`<@`、`||`，下拉框同时支持自由输入任意 PG 操作符。

**注意事项**：
- `GIST + = / <>` 需要 PG 启用 `btree_gist` 扩展（`CREATE EXTENSION btree_gist`）。UI 会即时提示。
- EXCLUDE 在 PG 内部会创建一个隐式 GIST / SPGIST 索引，但**不会**出现在「索引」Tab；逆向导入也会自动跳过。

SQL 输出示例：

```sql
CREATE TABLE bookings (
  id      BIGINT      PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  room_id INT         NOT NULL,
  during  TSTZRANGE   NOT NULL,
  active  BOOLEAN     NOT NULL DEFAULT TRUE,
  CONSTRAINT bookings_no_overlap
    EXCLUDE USING GIST (room_id WITH =, during WITH &&)
    WHERE (active)
);
```

## 7. 索引（Phase 8 重写）

「索引」Tab。每个索引卡片显示：`UNIQUE` 标签、索引类型标签（`BTREE / GIN / ...`）、名称、列预览、`WHERE` 谓词。

点 `+ 添加索引` 进入 Modal：

| 字段 | 说明 |
|------|------|
| 索引名 | 留空 → `idx_<table>_<col1>_<col2>...`（UNIQUE 时前缀 `uq_`） |
| 索引类型 | `BTREE / HASH / GIN / GIST / BRIN / SPGIST` |
| UNIQUE | 开关 |
| 索引列 | 至少 1 行；每行可切换"字段/表达式"模式，可上下移动 |
| WHERE 谓词 | 部分索引子句（裸表达式） |

**列编辑器每行的字段**：

- **模式**：`字段`（下拉选）/ `表达式`（如 `LOWER(name)`、`(payload->>'kind')`）
- **方向**：`ASC` / `DESC`（ASC 是默认值，不输出到 SQL）
- **opclass**：可选输入（如 `jsonb_path_ops`、`gin_trgm_ops`）

校验：至少 1 列；每行必须填字段或表达式之一；同一字段不能出现两次。

输出示例：

```sql
-- 多列 + 部分索引
CREATE INDEX idx_models_visible    ON models (enabled, status)         WHERE deleted_at IS NULL;
CREATE INDEX idx_models_sort_order ON models (sort_order ASC, id ASC)  WHERE deleted_at IS NULL;
-- 表达式索引
CREATE INDEX idx_models_name_lower ON models (LOWER(name))             WHERE deleted_at IS NULL;
-- GIN + opclass + JSONB
CREATE INDEX idx_conversation_messages_content
  ON conversation_messages USING GIN (content jsonb_path_ops);
```

> 删除一个索引列绑定的字段后，索引会失效但**不会**自动联动删除；下次执行 SQL 时 PG 会报错，回到 UI 修复即可。这是有意为之，避免静默丢配置。

## 9. SQL 生成与直接执行

### 9.1 预览 / 复制 / 下载

`工具栏 → SQL 预览`：

- 按分节展示：`-- ENUM 类型`、`-- 数据表`、`-- 外键约束`、`-- 索引`、`-- 注释`。
- 顶部「复制」按钮：拷到剪贴板。
- 「下载 .sql」：浏览器下载为 `<project>.sql`。

### 9.2 执行 DDL 到当前数据库

需要先「连接」。`工具栏 → 执行 DDL`：

- **事务模式**（默认）：所有语句包在一个 `BEGIN ... COMMIT`；任一失败整体 ROLLBACK。适合首次建表。
- **逐条模式**：每条独立提交，出错后继续后面的语句。适合"补丁式"增量执行。

弹窗下方实时显示每条语句的执行状态（✓ / ✗ + 错误详情）。

执行顺序固定：

```
1. CREATE TYPE (ENUMs)
2. CREATE TABLE (按外键依赖拓扑排序)
3. ALTER TABLE ADD CONSTRAINT FOREIGN KEY
4. CREATE INDEX
5. COMMENT ON ...
```

## 10. 从已有数据库导入

`工具栏 → 导入数据库结构`（需先连接，会提示"将覆盖当前设计"）：

后端读取 `information_schema` + `pg_catalog` 还原成 `ProjectFile`，覆盖前端当前项目。导入逻辑会自动跳过：

- `__dbdesign` 配置表（防止把自己导成普通表）。
- 主键 / UNIQUE 约束附带的隐式索引（已由列级 `isPrimaryKey / isUnique` 表达）。
- EXCLUDE 约束附带的隐式 GIST 索引（由约束分支表达）。
- PG 自动为 NOT NULL 列生成的 `*_not_null` CHECK。
- 单列 CHECK 中只引用本列的表达式（已由列级 `checkConstraint` 表达）。

**Phase 8 新增导入支持**：

- IDENTITY 列被识别为 `field.identity = 'ALWAYS' | 'BY DEFAULT'`，不再误判为 SERIAL。
- 索引完整保留方向 / opclass / 部分索引谓词；表达式索引整体存为单一 expression 列，下次再保存时按原文回放。
- EXCLUDE 约束被解析为 `TableConstraint.kind = 'EXCLUDE'`，含 USING / elements / WHERE / DEFERRABLE。

> 复杂表达式索引在 PG 不同版本下输出形态略有差异；本工具采用"原样保留 `pg_get_indexdef` 文本"的退化策略 —— 解析失败时不会丢索引，只是后续编辑变成"改字符串"。

## 11. 结构对比与 ALTER 生成

设计变更后，相对当前数据库的状态可生成增量 ALTER 脚本：

`工具栏 → 结构对比` → 弹出 Diff Modal：

- 列出 ENUM / 表 / 字段 / 索引 / 约束 / FK / 注释的所有变化（新增 / 删除 / 修改）。
- 底部给出可执行的 ALTER 脚本（含 `IF EXISTS` 兜底）。
- 「应用到数据库」按钮：直接调 `/api/schema/execute` 执行该脚本。

特殊规则：

- **同名索引** 的列 / 方向 / opclass / WHERE / INCLUDE / UNIQUE / 类型任一变更 → 输出 `DROP INDEX + CREATE INDEX`（PG 不支持 ALTER INDEX 改结构）。
- **同名 EXCLUDE 约束** 的任一字段变更 → `DROP CONSTRAINT + ADD CONSTRAINT`。
- **ENUM** 仅支持 `ADD VALUE`；删除值需用户手工迁移。
- **列类型变更** 输出 `ALTER COLUMN ... TYPE ... USING col::new_type`，复杂场景需手改 USING 子句。

## 12. 撤销 / 重做 / 快捷键

| 键位 | 作用 |
|------|------|
| `Ctrl/Cmd + S` | 保存到 `.dbdesign.json` |
| `Ctrl/Cmd + Z` | 撤销（数据结构层面）|
| `Ctrl/Cmd + Shift + Z` | 重做 |
| `Delete` | 删除当前选中表（带二次确认）|
| `Escape` | 关闭当前 Modal / 取消单元格编辑 |

撤销栈（zundo）仅记录数据结构变更（表、字段、ENUM、索引、约束等）。**画布位置 / 视口变更不入栈**，避免拖动产生几十次冗余历史。

## 13. 数据持久化与安全

| 数据 | 存储位置 | 生命周期 |
|------|----------|----------|
| 项目结构（`ProjectFile`） | `.dbdesign.json` 文件 / 目标库 `__dbdesign` 表 | 显式保存才落盘 |
| 连接配置（含密码） | 浏览器 `sessionStorage` | 关闭标签即清空 |
| 撤销历史 | 浏览器内存 | 关闭页面即清空 |
| 服务端会话 | **无** | 每请求建池-执行-释放 |

服务端没有数据库，所有"配置"都跟着请求或文件走，方便部署到内网共享服务器供多人使用。

## 14. 典型工作流

### 14.1 从零开始设计

```
新建项目 → 字段表加列 → 用预设填默认值
     ↓
约束 Tab 加跨列约束 → 索引 Tab 加部分索引
     ↓
关系图 → 整理布局
     ↓
连接本地 PG → SQL 预览 → 执行 DDL（事务模式）
     ↓
存库 → 关掉浏览器
```

### 14.2 拿到一个已有库

```
连接 → 导入 → 关系图整理布局 → 保存为 .dbdesign.json（带 git 走）
```

### 14.3 增量改库

```
打开旧 .dbdesign.json → 改字段 / 加索引 / 加 EXCLUDE
     ↓
连接到目标库 → 结构对比 → 检查 ALTER 脚本
     ↓
逐条模式执行 → 存库 → commit .dbdesign.json
```

### 14.4 多人评审

```
开 .dbdesign.json → 关系图 → 全屏
     ↓
单击节点查 PK/FK → 双击节点跳设计器看字段细节
     ↓
评审完不动改动即可关闭；如调整过画布视口会标 dirty，可丢弃
```

## 15. 常见问题

**Q：保存按钮灰着？**
还没新建/打开项目；或浏览器不支持 File System Access API（Safari/Firefox 会回退为下载方式）。

**Q：导入后某列被识别为 `USER-DEFINED` 但找不到对应 ENUM？**
说明该列引用了别的 schema 的类型而 inspector 没扫描到 → 在新建 ENUM 时手动补一个同名 `schema.name`，或把列改回内建类型。

**Q：执行 DDL 报 `extension btree_gist does not exist`？**
你写了 EXCLUDE 用 GIST 且包含 `=` / `<>` 操作符；先在目标库执行 `CREATE EXTENSION btree_gist;` 再来。

**Q：执行 DDL 报 `pgcrypto`（gen_random_uuid 找不到）？**
PG13- 需要 `CREATE EXTENSION pgcrypto;`；PG14+ 自带 `gen_random_uuid()`。或改用 `uuid_generate_v4()`（需 `uuid-ossp` 扩展）。

**Q：撤销没把节点位置一起回滚？**
按设计：布局变更不进入撤销栈。想还原布局只能手动拖、点「整理布局」或重新打开文件。

**Q：旧 `.dbdesign.json` 打不开？**
Phase 8 兼容 Phase 7 的 `fieldIds` 索引格式（自动迁移）。如果文件是手工编辑过且违反 schema 约束，可能解析失败 → 在 DevTools 控制台看具体报错。

**Q：「读库」覆盖了我没保存的设计？**
读库前会弹确认；接受后会用 DB 内 `__dbdesign` 行覆盖前端项目。下次注意"先保存再读库"。

**Q：怎么把字段定义为数组（如 `INTEGER[]`）？**
目前**没有 UI** 暴露"数组"开关。数据模型层面 `FieldDefinition.isArray` 已就绪、SQL 生成器和反向导入也都能识别 PG 数组（`_int4[]` 等），但需要手动编辑 `.dbdesign.json` 把对应字段加 `"isArray": true`，或从已有库导入再保存。后续版本会补 UI。
