# 阶段 1（首批）— 扩充类型库 + 生成列（可执行方案）

> 日期：2026-06-11
> 来源：`development-plan-20260611-coverage-gap.md` 阶段 1 中侵入最小、体感最直接的两项。
> 范围：A 区（扩充字段类型库 + 时间精度）+ B 区（生成列 `GENERATED ALWAYS AS (expr) STORED`）。
> 不在范围：复合外键、`CREATE SCHEMA` 输出、扩展管理（同属阶段 1，但属破坏性/跨对象改动，单独排期）。
> 设计原则：**不改 `version: '1.0'`**。新增字段全部可选，老 `.dbdesign.json` 直接兼容。

## 目标 SQL 形态（验收基线）

完成后系统必须能产出：

```sql
-- 1. 二进制 / 网络 / 位 / 全文 / XML
CREATE TABLE assets (
  id        BIGINT       PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  payload   BYTEA,
  client_ip INET,
  net       CIDR,
  flags     BIT(8),
  permbits  BIT VARYING(64),
  search    TSVECTOR,
  doc       XML,
  amount    MONEY
);

-- 2. 时间精度
CREATE TABLE events (
  happened_at TIMESTAMP(3)   NOT NULL,
  duration    INTERVAL(6),
  day_window  DATERANGE
);

-- 3. 区间类型 + EXCLUDE（与现有 EXCLUDE 能力天然衔接）
CREATE TABLE bookings (
  room_id INT       NOT NULL,
  during  TSTZRANGE NOT NULL,
  CONSTRAINT bookings_no_overlap EXCLUDE USING GIST (room_id WITH =, during WITH &&)
);

-- 4. 生成列
CREATE TABLE rectangles (
  width  NUMERIC NOT NULL,
  height NUMERIC NOT NULL,
  area   NUMERIC GENERATED ALWAYS AS (width * height) STORED
);
```

## A 区 — 扩充类型库

### A1. `typeDefinitions.ts` — 类型分组 + 元数据

**现状**：`PG_TYPE_GROUPS`（`typeDefinitions.ts:8`）只有 5 组；`TypeGroup` 项仅 `hasLength?` / `hasPrecision?` 两个标志（`typeDefinitions.ts:5`）。`typeHasPrecision` 当前同时驱动「精度 + 小数」两个输入框。

**改动**：

1. `TypeGroup` 项扩展元数据，把「精度」与「小数」拆开，并允许定制上限：

```ts
types: {
  value: PgFieldType;
  label: string;
  hasLength?: boolean;     // VARCHAR/CHAR/BIT/BIT VARYING
  hasPrecision?: boolean;  // 单参精度：NUMERIC / TIME* / INTERVAL
  hasScale?: boolean;      // 第二参小数：仅 NUMERIC
  maxPrecision?: number;   // NUMERIC=1000；时间类=6
}[];
```

2. `NUMERIC` 改为 `hasPrecision + hasScale, maxPrecision: 1000`（行为不变）。

3. 新增分组与类型（`value` 用 PG 规范名，直接进 `renderFieldType` 的 `${type}(...)`）：

| 分组 | 新增类型 |
|---|---|
| 数值类型（追加） | `MONEY` |
| 日期时间（追加精度） | `TIME`/`TIMESTAMP`/`TIMESTAMPTZ` 加 `hasPrecision, maxPrecision: 6`；新增 `TIMETZ`、`INTERVAL`（均 `hasPrecision, maxPrecision: 6`） |
| 二进制 / 网络 / 位（新） | `BYTEA`、`INET`、`CIDR`、`MACADDR`、`MACADDR8`、`BIT`(hasLength)、`BIT VARYING`(hasLength) |
| 区间类型（新） | `INT4RANGE`、`INT8RANGE`、`NUMRANGE`、`TSRANGE`、`TSTZRANGE`、`DATERANGE` |
| 全文 / 文档（新） | `TSVECTOR`、`TSQUERY`、`XML` |
| 几何类型（新，可选） | `POINT`、`LINE`、`LSEG`、`BOX`、`PATH`、`POLYGON`、`CIRCLE` |

4. 新增 `typeHasScale(type)`，与现有 `typeHasLength`/`typeHasPrecision`（`typeDefinitions.ts:67-73`）并列；并把 `getMaxPrecision(type)` 暴露给 UI。

### A2. `sqlGenerator.ts` / `schemaDiff.ts` — `renderFieldType`

**现状**（`sqlGenerator.ts:33-51`，`schemaDiff.ts:25-40` 有一份同源副本）：

```ts
} else if (typeHasLength(field.type) && field.length) {
  base = `${field.type}(${field.length})`;
} else if (typeHasPrecision(field.type) && field.precision) {
  base = field.scale != null ? `${type}(${p}, ${s})` : `${type}(${p})`;
}
```

**改动**：逻辑几乎不变（`scale == null` 时已天然产出单参 `TYPE(p)`，正好覆盖时间精度），只需两处健壮化——把 `&& field.length` / `&& field.precision` 改为 `!= null`，以支持 `TIME(0)` 这类合法的 0 值精度。两份副本同步改。

> 即：`MONEY`/`BYTEA`/`INET`/区间/全文等无参类型走 `else base = field.type` 分支，原样输出；`BIT(8)` 走 length 分支；`TIMESTAMP(3)`/`INTERVAL(6)` 走 precision 分支。**无需为新类型写专门逻辑**。

### A3. `FieldTypeSelect.tsx` — 输入框联动

**现状**：长度框（`FieldTypeSelect.tsx:51-61`）、精度+小数框（`64-85`）；`handleTypeChange`（`13-20`）切类型时清理关联属性。

**改动**：

- 精度框：`max` 由硬编码 `1000` 改为读 `getMaxPrecision(field.type)`。
- 小数框：渲染条件由 `typeHasPrecision` 改为 `typeHasScale`（时间精度类只剩单框）。
- `handleTypeChange`：`if (!typeHasScale(type)) changes.scale = undefined;`，避免从 NUMERIC 切到 TIMESTAMP 残留 scale。
- 长度框 `max`：BIT 可放宽（如 10^7），VARCHAR 维持。

### A4. 逆向导入 `schemaImporter.ts` + 后端 inspector

**现状**：`DATA_TYPE_MAP`（`schemaImporter.ts:23`）/`ARRAY_UDT_MAP`（`50`）；后端列查询的类型 `CASE`（`schemaInspector.ts:252-260`）对 `USER-DEFINED` 返回 `udt_name`。

**改动**：

1. `DATA_TYPE_MAP` 补全新类型映射（`information_schema.data_type` 名 → 内部枚举）：
   - `bytea→BYTEA`、`interval→INTERVAL`、`money→MONEY`、`xml→XML`、`bit→BIT`、`bit varying→BIT VARYING`、`tsvector→TSVECTOR`、`tsquery→TSQUERY`、`inet→INET`、`cidr→CIDR`、`macaddr→MACADDR`、`macaddr8→MACADDR8`、`time with time zone→TIMETZ`。
2. **区间/几何类型的 wrinkle**：它们经 `data_type='USER-DEFINED'` → inspector 回 `udt_name`（`int4range`/`tstzrange`/`point`…，小写）。在 `DATA_TYPE_MAP` 加这些小写键（`int4range→INT4RANGE` …），使 `parseType`（`schemaImporter.ts:79`）不再误判为 ENUM 引用。
3. `parseType` 的时间精度解析：`time(3)`/`timestamp(3)`/`interval(6)` 经后端 CASE 已带 `(p)`？—— 当前 CASE 只对 `character_maximum_length` 与 `numeric` 追加括号，**时间精度未追加**。需在 `schemaInspector.ts` 的 CASE 补 `datetime_precision`（来自 `information_schema.columns.datetime_precision`）分支，否则导入丢精度。`parseType` 对应放开 `args[0]→precision`（非 NUMERIC 的单参场景）。
4. `ARRAY_UDT_MAP`（`50`）按需补 `_bytea`/`_inet`/`_int4range` 等高频数组元素类型（可后续按反馈增量）。

### A5. 测试

- `sqlGenerator.test.ts`：新增「无参类型原样输出」（`BYTEA`/`INET`/`TSTZRANGE`）、`BIT(8)`、`TIMESTAMP(3)`、`TIME(0)`（验证 0 精度不被吞）。
- `schemaImporter.test.ts`：新增 `interval`、`int4range`（走 USER-DEFINED→udt_name 路径不再误判 ENUM）、`bit varying(64)` 往返。

## B 区 — 生成列（GENERATED ALWAYS AS … STORED）

### B1. `types/schema.ts` — 数据模型

**现状**：`FieldDefinition`（`schema.ts:42-61`）有 `identity?: IdentityKind`（`60`）、`defaultValue?`、`checkConstraint?`。

**改动**：新增可选字段，与 `identity` 对称：

```ts
export interface FieldDefinition {
  // ... 现有字段
  /** GENERATED ALWAYS AS (<expr>) STORED；与 identity / defaultValue 三者互斥 */
  generatedExpression?: string;   // 仅表达式本体，不含 GENERATED…STORED 外壳
}
```

PG 当前仅支持 `STORED`（无 `VIRTUAL`），故不需额外 `stored` 标志。

### B2. `sqlGenerator.ts` — 渲染

**现状** `renderColumnDefinition`（`sqlGenerator.ts:56-78`）优先级：`identity > defaultValue`。

**改动**：插入最高优先级分支（`schemaDiff.ts:626` 的 `renderColumnInline` 同源副本同步）：

```ts
if (field.generatedExpression?.trim()) {
  parts.push(`GENERATED ALWAYS AS (${field.generatedExpression.trim()}) STORED`);
} else if (field.identity) {
  parts.push(`GENERATED ${field.identity} AS IDENTITY`);
} else if (field.defaultValue ...) {
  ...
}
```

NOT NULL 既有逻辑保留（生成列可声明 NOT NULL，输出顺序 `... NOT NULL GENERATED ALWAYS AS (...) STORED` 合法）。

### B3. `ConstraintConfig.tsx` — 编辑器

**现状**：列约束 Modal 已含 CHECK + IDENTITY（`ConstraintConfig.tsx`，IDENTITY 区 `91-122`，仅 `SMALLINT/INTEGER/BIGINT` 显示）。

**改动**：

- 在 IDENTITY 区下方加「生成列（GENERATED）」`Input.TextArea`（等宽字体，占位 `width * height`），**所有类型可见**。
- 互斥联动（`handleSave`，`37-49`）：
  - 填了 `generatedExpression` → 清空 `identity` 与 `defaultValue`。
  - `identity !== 'NONE'` → 清空 `generatedExpression`。
- 提示文案：生成列与 IDENTITY/默认值互斥；表达式只能引用**本表同行**的列。

### B4. `schemaDiff.ts` — 差异

**现状**：`normalizeField`（`schemaDiff.ts:58-68`）比较 type/nullable/default/comment/unique。

**改动**：

- `NormalizedField` + `FieldChange.changes` 加 `generated`。
- **限制（务必在 UI 文案点明）**：PG 无法把普通列原地 `ALTER` 成生成列（反之 `DROP EXPRESSION` PG13+ 才有）。因此生成列表达式变更一律按 **drop + add column** 处理（已有 `columnsDropped`/`columnsAdded` 通道天然支持），不发 `ALTER COLUMN`。在 diff 概要里对这类变更标注「将重建列（数据丢失风险）」。

### B5. 逆向导入 — inspector + importer

**现状**：`DbColumn`（`backend/src/types/index.ts:57`）有 `isIdentity`/`identityGeneration`；`fetchColumns`（`schemaInspector.ts:237`）SELECT 这些列；前端 `inspectionToProject`（`schemaImporter.ts:178-214`）映射。

**改动**：

1. 后端 `fetchColumns` 增 `c.is_generated`（`information_schema.columns.is_generated = 'ALWAYS'`）+ `c.generation_expression`；`DbColumn` 加 `isGenerated?: boolean` / `generationExpression?: string`。
2. 前端 importer：`isGenerated` 为真时置 `field.generatedExpression = generationExpression`，并清 `defaultValue`/`identity`（与现有 IDENTITY 处理对称，`schemaImporter.ts:204-208`）。

### B6. 测试

- `sqlGenerator.test.ts`：生成列输出 `GENERATED ALWAYS AS (width * height) STORED`；与 identity/default 同设时优先级正确（generated 胜出，且不输出 DEFAULT）。
- `schemaDiff.test.ts`：普通列→生成列、生成列表达式变更，均产出 drop+add 而非 ALTER。

1. `npm run dev` → 新建表，逐一选用 `BYTEA`/`INET`/`BIT(8)`/`TSTZRANGE`/`TIMESTAMP(3)`/`MONEY`，SQL 预览逐条核对目标基线。

3. 往返：对一个含 `interval`/`bit varying`/`int4range`/生成列的真实库执行「逆向导入」→ 字段类型/精度/生成表达式正确还原，再「SQL 预览」与原库一致。
4. 向后兼容：用旧 `.dbdesign.json`（无 `generatedExpression`）打开无报错。
5. `npm run test` 全绿（A 区零破坏；B 区新增字段不影响既有用例）。
