# Phase 8 — 索引高级特性 + 列定义增强 + EXCLUSION 约束

> 日期：2026-06-05
> 范围：A 区（索引）+ B 区（IDENTITY 列、默认值预设）+ 约束扩展（EXCLUSION 排斥约束）
> 背景：对账 `refe/migrations/001-005` 后发现，系统已能产出 80% 的结构 DDL，但索引层和 IDENTITY 列是 1:1 复刻这套 migrations 的最大短板。同时 Phase 7 把 EXCLUSION 显式推迟，至今仍是初始需求中的最后一项空白，本期一并补齐。
> 不在范围：触发器 / 函数（C 区）、种子 DML（C 区）、多文件迁移管理（D 区）—— 见 `development-plan-20260605-phase9.md`。

## 目标 SQL 形态（验收基线）

以下三条 DDL 是本 Phase 完成后系统必须能产出的代表样本：

```sql
-- 1. 多列 + 部分索引
CREATE INDEX idx_models_visible    ON models (enabled, status)         WHERE deleted_at IS NULL;
CREATE INDEX idx_models_sort_order ON models (sort_order ASC, id ASC)  WHERE deleted_at IS NULL;

-- 2. 表达式索引 + 部分索引
CREATE INDEX idx_models_name_lower ON models (LOWER(name))             WHERE deleted_at IS NULL;

-- 3. GIN + opclass + JSONB
CREATE INDEX idx_conversation_messages_content
  ON conversation_messages USING GIN (content jsonb_path_ops);

-- 4. IDENTITY 列
CREATE TABLE upload_tasks (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  ...
);

-- 5. 列默认值常用函数
CREATE TABLE users (
  id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  ...
);

-- 6. EXCLUSION 排斥约束（典型场景：会议室预订防重叠）
CREATE TABLE bookings (
  id      BIGINT      PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  room_id INT         NOT NULL,
  during  TSTZRANGE   NOT NULL,
  active  BOOLEAN     NOT NULL DEFAULT TRUE,
  CONSTRAINT bookings_no_overlap
    EXCLUDE USING GIST (room_id WITH =, during WITH &&)
    WHERE (active)
);
```

## A 区 — 索引

### A1. 数据模型重构（`types/schema.ts`）

将 `IndexDefinition.fieldIds: string[]` 升级为结构化的 `columns`：

```typescript
export type IndexType = 'BTREE' | 'HASH' | 'GIN' | 'GIST' | 'BRIN' | 'SPGIST';
export type IndexDirection = 'ASC' | 'DESC';
export type IndexNullsOrder = 'FIRST' | 'LAST';

export interface IndexColumn {
  // 二选一：引用字段，或自由表达式
  fieldId?: string;
  expression?: string;            // 如 "LOWER(name)"、"(payload->>'kind')"

  direction?: IndexDirection;     // 默认 ASC
  opclass?: string;               // 如 'jsonb_path_ops'、'gin_trgm_ops'
  nulls?: IndexNullsOrder;        // 默认按 PG 推断；显式输出时附加 NULLS FIRST/LAST
}

export interface IndexDefinition {
  id: string;
  name: string;                   // 可空字符串则按 idx_<table>_<col1>_<col2>... 生成
  columns: IndexColumn[];         // ← 新结构，取代 fieldIds
  isUnique: boolean;
  indexType?: IndexType;          // 默认 BTREE
  predicate?: string;             // 部分索引 WHERE 子句（不含 WHERE 关键字）
  include?: string[];             // INCLUDE 覆盖列（字段名数组；为后续覆盖索引留口）
  comment?: string;
}
```

**JSON 向后兼容**：`projectStore.loadProject` 在反序列化时拦截一次：

```typescript
function migrateIndex(idx: any): IndexDefinition {
  if (Array.isArray(idx.fieldIds)) {
    return {
      id: idx.id,
      name: idx.name,
      isUnique: !!idx.isUnique,
      indexType: idx.indexType,
      columns: idx.fieldIds.map((fid: string) => ({ fieldId: fid, direction: 'ASC' as const })),
    };
  }
  return idx as IndexDefinition;
}
```

旧 `.dbdesign.json` 加载后立刻被规范化；保存时只写新结构。

### A2. Store 扩展（`projectStore.ts`）

```typescript
addIndex:    (tableId: string, index: Omit<IndexDefinition, 'id'>) => string;
updateIndex: (tableId: string, indexId: string, changes: Partial<IndexDefinition>) => void;
deleteIndex: (tableId: string, indexId: string) => void;
```

`updateIndex` 是**本期新增**——之前 store 只有 `add/delete`，缺改的能力。所有变更都置 `isDirty = true` 并进 zundo 撤销栈。

### A3. UI — `TableIndexesPanel`（新组件）

放在 `TableEditor` 内，与「约束」并列的一个 Tab，沿用 `TableConstraintsPanel` 的视觉风格：

**列表项展示**：
- 类型 Tag：`BTREE` / `GIN` / ...（`isUnique` 时前面再加 `UNIQUE` Tag，蓝色）
- 索引名（留空显示「(自动命名)」）
- 列预览：`(col_a ASC, LOWER(col_b))` 单色等宽字体
- `WHERE 谓词` 缩略：用 ellipsis 限宽
- 操作：编辑 / 删除

**添加 / 编辑 Modal**：
- 索引名（可选）
- 索引类型下拉
- 唯一开关
- **列编辑器**（核心，行可上下移动）：每行：
  - 模式切换：「字段」/「表达式」（互斥）
  - 字段模式：字段下拉
  - 表达式模式：单行输入框，等宽字体（输入 `LOWER(name)` 或 `(payload->>'kind')`）
  - 方向：ASC / DESC（默认 ASC）
  - opclass：可选输入（占位提示 `jsonb_path_ops`）
- WHERE 谓词：TextArea，可选
- 校验：至少 1 列；每行必须填字段或表达式之一；字段行不允许重复同一字段（同一字段两次方向不同的情况罕见，暂禁）

**列表头按钮**：`+ 添加索引`。

### A4. SQL 生成（`sqlGenerator.ts`）

替换 `generateIndexStatements`：

```typescript
function renderIndexColumn(col: IndexColumn, fields: FieldDefinition[]): string {
  const head = col.fieldId
    ? quoteIdent(fields.find(f => f.id === col.fieldId)?.name ?? '?')
    : (col.expression ?? '').trim();
  const parts = [head];
  if (col.opclass) parts.push(col.opclass);
  const tail: string[] = [];
  if (col.direction === 'DESC') tail.push('DESC');
  // ASC 是默认值，不显式输出，保持 SQL 简洁
  if (col.nulls) tail.push(`NULLS ${col.nulls}`);
  return tail.length ? `${parts.join(' ')} ${tail.join(' ')}` : parts.join(' ');
}

function renderCreateIndex(idx: IndexDefinition, table: TableDefinition): string {
  const name = idx.name?.trim() || autoIndexName(idx, table);
  const using = idx.indexType && idx.indexType !== 'BTREE' ? ` USING ${idx.indexType}` : '';
  const cols = idx.columns.map(c => renderIndexColumn(c, table.fields)).join(', ');
  const include = idx.include?.length ? ` INCLUDE (${idx.include.map(quoteIdent).join(', ')})` : '';
  const where = idx.predicate?.trim() ? ` WHERE ${idx.predicate.trim()}` : '';
  const unique = idx.isUnique ? 'UNIQUE ' : '';
  return `CREATE ${unique}INDEX ${quoteIdent(name)} ON ${qualified(table.schema, table.name)}${using} (${cols})${include}${where};`;
}
```

`autoIndexName` 规则：`idx_<table>_<col1>_<col2>...`；UNIQUE 时前缀 `uq_`；表达式列名用其首词或哈希。

### A5. 逆向导入（`backend/services/schemaInspector.ts`）

扩展索引查询，读 `pg_index` + `pg_get_indexdef`：

```sql
SELECT
  c.relname                                            AS name,
  i.indisunique                                        AS is_unique,
  am.amname                                            AS index_type,
  pg_get_indexdef(i.indexrelid, 0, true)               AS definition,
  pg_get_expr(i.indpred, i.indrelid, true)             AS predicate,
  i.indkey                                             AS column_attnums,
  i.indoption                                          AS column_options
FROM pg_index i
JOIN pg_class c   ON c.oid = i.indexrelid
JOIN pg_class t   ON t.oid = i.indrelid
JOIN pg_am   am   ON am.oid = c.relam
JOIN pg_namespace n ON n.oid = t.relnamespace
WHERE n.nspname = $1 AND t.relname = $2
  AND NOT i.indisprimary                       -- 主键索引由 PK 表达
  AND NOT EXISTS (                             -- 跳过被 UNIQUE 约束实现的索引（约束分支已处理）
    SELECT 1 FROM pg_constraint con
    WHERE con.conindid = i.indexrelid AND con.contype IN ('p', 'u')
  );
```

**解析策略**：直接用 `pg_get_indexdef` 返回的字符串做最简单的二选一：
- 如果索引所有列都是简单字段引用 → 把每个 `indkey` 映射回字段 ID，方向从 `indoption` 位提取
- 如果出现表达式 / opclass → 退化为「整体保存原始 def 作为一个 expression 索引」，下次 SQL 生成时直接吐 `pg_get_indexdef` 的内容（外面再裹 `CREATE INDEX` 是冗余的，所以保留 def 原文，跳过自渲染）

实施上为简化先走第二条路：**任何复杂索引一律以 `IndexDefinition.columns = [{ expression: '<整体表达式>'}]` 形式回存**。后续如需精细 round-trip 再升级。

`__dbdesign` 与主键索引继续过滤。

### A6. Diff（`schemaDiff.ts`）

`indexesAdded` / `indexesDropped` 已有。本期增加：

- **同名索引定义变更检测**：按 `name` 匹配后，对比 `columns / isUnique / indexType / predicate / include` 任一不同 → drop + add（PG 不支持 ALTER INDEX 改结构）。
- 新表内联：索引始终独立于 `CREATE TABLE` emit，不进 `CREATE TABLE` 块；现状已如此。

## B 区 — 列定义增强

### B1. IDENTITY 列

**数据模型**：

```typescript
export type IdentityKind = 'ALWAYS' | 'BY DEFAULT';

export interface FieldDefinition {
  // ... 已有字段
  identity?: IdentityKind;       // 仅对 INTEGER / BIGINT / SMALLINT 有意义
}
```

**UI**：FieldsTable 的「类型」列保留 INT/BIGINT 不变；在「默认值」列旁不增列，而是在 `ConstraintConfig`（已有的小弹窗）里加一组单选：
- `无（普通列）`
- `GENERATED BY DEFAULT AS IDENTITY`（推荐，可手动覆盖）
- `GENERATED ALWAYS AS IDENTITY`（严格，禁止手动赋值）

仅当类型 ∈ {SMALLINT, INTEGER, BIGINT} 时显示该单选。选中后**强制清空** `defaultValue`（identity 与 default 互斥）。

**SQL 生成**：在 `renderColumnDefinition` 中：

```typescript
if (field.identity) {
  // type 仍输出（INTEGER/BIGINT），尾部追加
  parts.push(`GENERATED ${field.identity} AS IDENTITY`);
  // 跳过 DEFAULT / NULL 输出（NOT NULL 由 IDENTITY 隐式保证，仍可显式输出 NOT NULL）
}
```

**与 SERIAL 的关系**：SERIAL/BIGSERIAL 保留为类型选项，作为「旧式快捷」。文案可以在类型下拉里给 SERIAL 加副标题「等价于 INTEGER + IDENTITY，新项目建议用 INTEGER + IDENTITY」。**不**自动迁移已有 SERIAL 字段。

### B2. 默认值预设

`FieldsTable` 的「默认值」列从单纯 `Input` 改为 `Input` + 右侧 `Dropdown` 触发器（Ant Design 的 `Input` 自带 `addonAfter`，放一个小图标按钮）：

预设清单（点击直接填入输入框，仍可手改）：

| 类别 | 值 |
|---|---|
| 时间戳 | `now()` / `CURRENT_TIMESTAMP` |
| UUID | `gen_random_uuid()` / `uuid_generate_v4()` |
| 布尔 | `true` / `false` |
| 数值 | `0` / `1` |
| 字符串 | `''` |
| JSON | `'{}'` / `'[]'` |
| 序列 | `nextval('seq_name')`（占位，需用户改名） |

实现上，预设是纯前端常量数组，类型敏感地分组高亮（例如类型 = TIMESTAMPTZ 时把「时间戳」组排在最前）。

## 约束扩展 — EXCLUSION 排斥约束

EXCLUSION 是 Phase 7 显式推迟的一项，也是初始需求清单中最后一个未实现的子点（见原始 `docs/plan/init.md` 与 `docs/plan/development-plan-20260605.md`）。本期把它接入既有的 `TableConstraint` 体系，与 UNIQUE / CHECK 并列。

### 选型理由

- **不新增顶层结构**：EXCLUSION 是约束，挂在表上、和约束名同命名空间，PG `pg_constraint.contype = 'x'`；放在 `TableConstraint` 里最贴 PG 自身的对象层级。
- **UI 入口与 UNIQUE / CHECK 共用**：用户在「约束」Tab 里一处管理所有 3 类约束，不再额外开 Tab。
- **逆向时与索引解耦**：尽管 EXCLUSION 底层会自动创建一个 GIST/SPGIST 索引，但逆向导入按 `pg_constraint` 分流，不让它出现在「索引」Tab。

### 数据模型扩展（`types/schema.ts`）

```typescript
// 已有：'UNIQUE' | 'CHECK'，本期追加 'EXCLUDE'
export type TableConstraintKind = 'UNIQUE' | 'CHECK' | 'EXCLUDE';

export type ExclusionIndexMethod = 'GIST' | 'SPGIST' | 'BTREE' | 'HASH';

/** EXCLUDE 子句中的一个元素：列/表达式 + 比较操作符 */
export interface ExclusionElement {
  // 二选一（与 IndexColumn 同形）
  fieldId?: string;
  expression?: string;
  /** WITH 后的操作符：'='、'<>'、'&&'、'@>'、'<@' 等；不带引号 */
  operator: string;
}

export interface TableConstraint {
  id: string;
  name?: string;
  kind: TableConstraintKind;

  // UNIQUE
  fieldIds?: string[];

  // CHECK
  expression?: string;

  // EXCLUDE —— 本期新增字段
  exclusionElements?: ExclusionElement[];
  exclusionUsing?: ExclusionIndexMethod;   // 默认 GIST
  exclusionWhere?: string;                 // 可选谓词，不含 WHERE 关键字与外层括号
  exclusionDeferrable?: boolean;           // 可选：DEFERRABLE
  exclusionInitiallyDeferred?: boolean;    // 配合 DEFERRABLE 使用

  comment?: string;
}
```

**字段命名**：所有 EXCLUDE 专用字段加 `exclusion` 前缀，避免与 UNIQUE / CHECK 的 `fieldIds / expression` 字段产生 union 类型歧义。

### Store 改动（`projectStore.ts`）

已有的 `addTableConstraint / updateTableConstraint / deleteTableConstraint` 三个 action 直接复用，无需新增。验证逻辑（kind === 'EXCLUDE' 时 `exclusionElements.length >= 1` 且每个 element 必须有 `fieldId` 或 `expression` 之一 + 非空 `operator`）放在 Modal 表单校验层，不下沉到 store。

### UI — `TableConstraintsPanel` 扩展

复用现有面板：

- 顶部按钮新增第三个：`+ 添加 EXCLUDE`（与 `+ 添加 UNIQUE` / `+ 添加 CHECK` 并排）
- 列表项 Tag 颜色：UNIQUE = 绿、CHECK = 紫、**EXCLUDE = 橙**（区分度高，且语义上表示"排斥/冲突防护"）
- 列表项内容预览：`USING GIST (room_id WITH =, during WITH &&) WHERE (active)`，等宽字体 + ellipsis 限宽

**编辑 / 添加 Modal**（`ConstraintEditModal` 内增加 EXCLUDE 分支）：

- 名称（可选，留空时自动生成 `ex_<table>_<col1>_<col2>`，与 `uq_` / `chk_` 风格一致）
- 索引方法下拉：默认 GIST；可选 SPGIST / BTREE / HASH
- **元素编辑器**（行可上下移动，至少 1 行）：每行：
  - 模式切换：「字段」/「表达式」（互斥，与索引列编辑器同形）
  - 字段模式：字段下拉
  - 表达式模式：单行输入框，等宽字体
  - 操作符：下拉 + 自由输入（常用预设：`=`、`<>`、`&&`、`@>`、`<@`、`||` 等，但允许用户输入任意 PG 操作符）
- WHERE 谓词：TextArea，可选
- 高级折叠面板：DEFERRABLE / INITIALLY DEFERRED 两个复选框
- Modal 标题：`添加 EXCLUDE 约束` / `编辑 EXCLUDE 约束`，与 UNIQUE / CHECK 风格一致

### SQL 生成

EXCLUDE 内联在 `CREATE TABLE` 列定义之后，与 UNIQUE / CHECK 并列：

```typescript
function renderExclusionConstraintInline(c: TableConstraint, table: TableDefinition): string {
  const name = c.name?.trim() || defaultExclusionName(c, table);
  const using = c.exclusionUsing ?? 'GIST';
  const elements = (c.exclusionElements ?? []).map(el => {
    const head = el.fieldId
      ? quoteIdent(table.fields.find(f => f.id === el.fieldId)?.name ?? '?')
      : (el.expression ?? '').trim();
    return `${head} WITH ${el.operator}`;
  }).join(', ');
  const where = c.exclusionWhere?.trim() ? ` WHERE (${c.exclusionWhere.trim()})` : '';
  const deferrable = c.exclusionDeferrable
    ? (c.exclusionInitiallyDeferred ? ' DEFERRABLE INITIALLY DEFERRED' : ' DEFERRABLE')
    : '';
  return `CONSTRAINT ${quoteIdent(name)} EXCLUDE USING ${using} (${elements})${where}${deferrable}`;
}
```

把 `renderTableConstraintInline` 改为按 `kind` 分发到 UNIQUE / CHECK / EXCLUDE 三个 renderer。

### 逆向导入（`backend/services/schemaInspector.ts`）

`pg_constraint` 查询过滤升级：`contype IN ('u', 'c', 'x')`。对 `contype = 'x'` 的行：

```sql
SELECT
  con.conname             AS name,
  con.contype             AS kind,        -- 'x'
  pg_get_constraintdef(con.oid) AS def,   -- 形如 'EXCLUDE USING gist (room_id WITH =, during WITH &&) WHERE (active)'
  am.amname               AS index_method
FROM pg_constraint con
JOIN pg_class      rel ON rel.oid = con.conrelid
JOIN pg_namespace  ns  ON ns.oid  = rel.relnamespace
LEFT JOIN pg_class idx ON idx.oid = con.conindid
LEFT JOIN pg_am    am  ON am.oid  = idx.relam
WHERE ns.nspname = $1 AND rel.relname = $2 AND con.contype = 'x';
```

**解析策略**：与 Phase 8 索引逆向同思路 —— 用 `pg_get_constraintdef` 返回的字符串做最简映射：

1. 简单情形：所有元素都是 `<col_name> WITH <op>` 形式 → 解析回字段 ID + 操作符
2. 复杂情形（含表达式）：退化为单一 expression 元素，保留原始片段，下次输出时直接重放

`InspectSchemaResponse.tables[i].constraints` 已有结构无需大改，新增 `kind = 'EXCLUDE'` 分支即可。

### Diff（`schemaDiff.ts`）

`tableConstraintsAdded / tableConstraintsDropped` 已有；本期：

- 按 `name` 匹配后比对 EXCLUDE 的所有字段（`exclusionElements / exclusionUsing / exclusionWhere / exclusionDeferrable*`）
- 任一不同 → drop + add（PG 不允许 ALTER EXCLUDE 约束改定义）

### 关系图（可选小改）

EXCLUDE 通常引用 1 个等值字段 + 1 个范围字段，关系图中无须新增边类型。可在 `TableNode` 表头那枚 🔒 徽标的 Tooltip 列表里追加该表的 EXCLUDE 约束名（与 UNIQUE / CHECK 并列）。

### 与索引层的边界澄清

EXCLUDE 约束在 PG 内部会自动创建一个底层索引（GIST/SPGIST 等），但：

- **不**在「索引」Tab 显示该自动索引
- **不**进 `IndexDefinition` 数据模型
- 逆向 `schemaInspector` 的索引查询要 `NOT EXISTS (SELECT 1 FROM pg_constraint con WHERE con.conindid = i.indexrelid AND con.contype IN ('p','u','x'))`，避免和 EXCLUDE 约束的自动索引重复

## 风险与权衡

| 风险 | 应对 |
|---|---|
| 表达式索引解析复杂、跨版本 PG 行为不一致 | 逆向时直接保留 `pg_get_indexdef` 原文作为单一 `expression`，避免过度解析 |
| IDENTITY 与 SERIAL 语义重叠，用户混淆 | 类型下拉里给 SERIAL 加「等价于 IDENTITY」副标题；不强制迁移，文档化推荐 |
| 旧 JSON 兼容性 | 在 `loadProject` 入口拦截，且在导入完成后立即触发 `setDirty(true)`，引导用户重新保存为新格式 |
| `IndexColumn.expression` 任意字符串，缺乏校验 | 不做静态分析，由 PG 执行报错；UI 仅非空校验。和 CHECK 表达式处理方式一致 |
| 多列索引方向粒度过细，UI 复杂 | 列编辑器只在「编辑」时展开方向 / opclass / NULLS；默认折叠为「字段 + ASC」单行，避免一开始就轰炸用户 |
| 部分索引谓词中引用的列被删除 | 不联动；执行 SQL 时由 PG 报错，回到 UI 修复（与 CHECK 表达式策略一致） |
| EXCLUDE 操作符自由输入，可能写错（如 `==`、`<>` 与 `!=` 混用） | UI 不做静态校验，由 PG 执行报错；下拉里只提供常用预设，自由输入需用户自负 |
| EXCLUDE USING 默认 GIST 但用户未装 `btree_gist`，等号场景会失败 | Modal 文案：当 USING=GIST 且任一元素操作符为 `=`/`<>` 时提示 "需启用 `btree_gist` 扩展"；不阻止生成 |
| EXCLUDE 自动索引被误识别为用户索引 | `schemaInspector` 索引查询附加 `NOT EXISTS (...con.conindid = i.indexrelid AND con.contype = 'x')`，本计划已纳入实现细则 |
| `kind = 'EXCLUDE'` 与原有 UNIQUE/CHECK 共用 `TableConstraint` 类型导致字段联合膨胀 | 接受单一结构体多 kind 的代价；TypeScript 校验下沉到 Modal 表单层，store 不做 discriminator 拆分 |
